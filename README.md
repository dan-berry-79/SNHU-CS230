# The Gaming Room Project Reflection

## Client and Requirements

The client for this project was The Gaming Room, a company seeking to expand its Draw It or Lose It game beyond its original Android platform. The goal was to design a scalable web-based application capable of supporting multiple teams, games, and users while operating across different platforms and devices. The project required consideration of software architecture, operating systems, memory management, storage management, distributed systems, and security.

## What I Did Well

One of the strongest aspects of my work was documenting and evaluating multiple operating system platforms before making a recommendation. I thoroughly compared Windows, Linux, macOS, and mobile platforms based on scalability, security, performance, and cost. This process helped me justify selecting Linux as the recommended server platform.

## Helpful Parts of the Design Process

Creating the software design document before implementation was particularly helpful because it required me to think through system requirements, architecture, and design decisions before writing code. The UML diagram and domain model helped visualize relationships between classes and provided a roadmap for implementation.

## What I Would Improve

If I were revising the project, I would expand the distributed systems and cloud architecture sections. While the design addresses scalability and cross-platform support, additional detail regarding cloud deployment strategies, load balancing, and redundancy would strengthen the document.

## Interpreting User Needs

Throughout the project, I focused on the client's requirements rather than technical preferences alone. The recommendation of a Linux-based cloud environment was based on the client's need for scalability, reliability, security, and cost-effective growth. Understanding user requirements is critical because successful software must solve business problems in addition to meeting technical objectives.

## Future Design Approach

When designing software in the future, I will continue using requirements analysis, UML modeling, domain modeling, and architectural evaluation before implementation begins. These techniques help identify potential issues early, improve communication among stakeholders, and create more maintainable software solutions.
